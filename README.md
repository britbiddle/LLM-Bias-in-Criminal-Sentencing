# Evaluating LLM Bias in Criminal Sentencing

**CS 2880: AI for Social Impact** · Team: MB Samuel, Brit Biddle, Lorraine Bichara

As AI systems are increasingly explored for legal applications — triaging cases to focus human judgment on the most complex ones, assisting with sentencing recommendations and legal research — understanding whether these systems encode societal biases becomes critical. In federal criminal sentencing, decades of research have documented disparities along racial and demographic lines, even after controlling for legally relevant factors like offense severity and criminal history. If large language models (LLMs) are trained on data that reflects these disparities, they may reproduce or even amplify them when applied to legal tasks.

This project investigates whether LLMs exhibit racial bias when asked to predict federal criminal sentences, and how that bias compares to documented disparities in actual sentencing outcomes. Using the JUSTFAIR dataset — ~600,000 federal sentencing records linked with defendant demographics, offense characteristics, and judge information — we design a matched-counterfactual ablation study that isolates the effect of demographic cues on LLM predictions across three distinct bias channels: explicit race labels, racially-associated names, and indirect socioeconomic proxies.

---

## Experimental design

**Data:** JUSTFAIR, filtered to federal drug trafficking cases with prison sentences and clean race coding. Final universe: 107,650 cases. Sample: 300 cases stratified by race (100 each White/Black/Hispanic), seed=42.

**Models (budget ~$10 API total):**
- OpenAI GPT-4o mini
- Anthropic Claude Haiku 4.5
- Qwen 2.5 7B (local, 4-bit quantized)
- Llama 3.2 3B (local, 4-bit quantized)

**Conditions (11):**

| Group | Conditions | Tests |
|---|---|---|
| `baseline` | Legally-relevant facts + sex only | Zero-demographic prior |
| `race_{W,B,H}` | + explicit race label | Direct surface bias |
| `race_debias_{W,B,H}` | + race label + "ignore demographics" instruction | Prompt-level mitigation effectiveness |
| `name_{W,B,H}` | + racially-associated defendant name | Implicit cue via name |
| `ses` | + education + age + dependents | Indirect bias via socioeconomic proxies |

**Prompt:** Structured fields in decoded English (not USSC codes). Includes the computed USSG guideline range from the Sentencing Table at (offense level × Criminal History Category). No directional/prescriptive language — statute citations (§4B1.1, 18 USC 924(c)) are kept as pure labels to avoid priming the model on sentence direction. Output is forced JSON `{reasoning, predicted_months}` at temperature 0.

**Analyses:**
1. Within-case matched counterfactual (paired t, Wilcoxon, TOST equivalence at ±5 mo, Cohen's d, Bonferroni-adjusted across the family of tests)
2. Unified OLS race coefficient — real judges vs race-matched vs debias-matched vs name-matched vs SES predictions
3. Mitigation test — does the debias instruction collapse the OLS coefficient?
4. Heterogeneous effects by discretion window, mandatory-minimum binding, offense severity
5. Per-case flip analysis, STATMIN anchoring, reasoning content, cross-model agreement, calibration/MAE
6. Prompt sensitivity appendix — two alternate templates (concise + verbose) on a 50-case subset

## How to run

The notebook is written for Google Colab with a T4 GPU runtime.

1. Open `notebooks/ablation_study.ipynb` in Colab.
2. Select Runtime → Change runtime type → T4 GPU.
3. Store your API keys in Colab Secrets (🔑 in the left sidebar):
   - `OPENAI_API_KEY`
   - `ANTHROPIC_API_KEY`
   - `HF_TOKEN` (required for gated Llama weights)
4. Upload `data/justfair_raw.csv` to the `LLM-Bias-in-Criminal-Sentencing` folder on your Google Drive (the notebook mounts Drive and reads from there).
5. Run cells top-to-bottom.
   - Section 1–6 set up data, decoders, and prompts.
   - Section 7 defines the model runners.
   - Section 8 is a smoke test (API models only, ~2 min) — verifies prompts and parsing before the full run.
   - Section 9 is the full run: 300 cases × 11 conditions × 4 models = 13,200 prompts. ~90 min wall-clock, ~$10 API cost. Resumable — saves incrementally to `results/study_v2_results.parquet` after each batch.
   - Section 10 runs the full analysis suite.
   - Section 11 runs the prompt-sensitivity appendix.

### Local run (outside Colab)

If you prefer running locally:
```bash
pip install -r requirements.txt
cp .env.example .env  # fill in your keys
```
Then uncomment the local-dev `ROOT = Path(...)` line in the setup cell and run. Note: local run requires a CUDA-capable GPU for the HF models (or skip them).

## Findings (headline)

The full study is described in the writeup (see `local/` for working drafts). High-level:

1. **Frontier LLMs are provably insensitive to explicit race labels** within ±5 months (TOST equivalence).
2. **But they reproduce real-world racial disparity structurally** — OLS race coefficients on LLM predictions track the coefficient on real judge sentences (~+9–13 months for Black defendants) even when race was never in the prompt.
3. **Prompt-level mitigation is insufficient** — telling the model to ignore demographics does not collapse the OLS coefficient.
4. **SES proxies leak racial information** — the structural disparity persists when only education/age/dependents are added to the prompt.

Implication: surface-level fairness audits (which show "swapping the race label doesn't change the output") are insufficient. Real legal deployments would propagate structural disparities even with demographic-blind prompts.
